# eijient-architect Skill

## 概要

「設計」と「実装」を**明確に分離**するスキル。実装前のシステムデザインだけを行い、
成果物はルートの `design-doc.md` 1つ。

```
探索 → 明確化質問 → 設計案2〜3案（推奨つき） → ユーザー選択 → design-doc.md 出力 → 停止
                                                                    ↓ 承認後
                       @eijient-architect-review でレビュー / @eijient で実装
```

**最重要ルール：**

1. **実装・編集は一切しない。** `design-doc.md` の Write だけが唯一の書き込み。
2. 設計案は列挙で終わらせない。**推奨案と理由を明確に述べる。**
3. 明確化質問への回答を得てから設計を確定する。勝手に仕様を決めない。
4. 証拠主義 — 既存構造への言及は file:line で引用する。推測で語らない。

## ワークフロー

### Step 1: 要求理解と現状確認

`git status` / `git log --oneline -30`、`CLAUDE.md` / `README` / 既存の設計ドキュメントを読む。

既存の `design-doc.md`（過去の設計）があれば、決定記録・Open Questions・Out-of-scope を
引き継ぐ。決定済みの再検討・却下案の再提案はしない。

### Step 2: 並列探索

Explore サブエージェント2〜3体で並列に調べ、メインコンテキストをクリーンに保つ：

- **①既存資産** — 類似実装・再利用できる関数/モジュール（エントリポイント・実行フロー・主要ファイルを file:line つきで）
- **②パターンと規約** — レイヤ構造・命名・エラーハンドリング・テストの流儀
- **③統合ポイント** — DB schema・認証・外部API・ジョブ・設定・デプロイ

サブエージェントの報告を鵜呑みにしない。設計の根拠になる重要主張は自分で
grep / 実ファイルでスポット検証する。

### Step 3: 明確化質問

設計を左右する不明点を質問にする。**少なく・重く・優先度順**：

- コードから判断できるものは質問にしない
- 各質問に「なぜ重要か / 回答によって設計のどこが変わるか」を含める
- セキュリティ・金銭・互換性に関わるものが先頭

**回答を得てから設計に進む。**

### Step 4: 設計案の比較

2〜3案を作る：**最小変更案** / **理想形案** / **実利バランス案**。
（案が実質1つしかない明白な設計では無理に複数作らず、その旨を明示して1案＋推奨理由で進める）

各案に書くこと：概要 / 変更ファイル数と新規コンポーネント数 / トレードオフ /
移行手順とロールバック。最後に**推奨案と理由を明確に**述べ、ユーザーに選択を求める。

### Step 5: design-doc.md の出力

採用案を `references/design-doc-template.md` に沿って詳細化し、ルート直下に **Write** する。

### Step 6: 停止して報告

```
📐 設計書を作成しました: design-doc.md

【採用した設計】
{1〜2行サマリー（却下案と却下理由も1行ずつ）}

【新規/変更コンポーネント】（{N}件）
1. {コンポーネント} — {担当ファイル範囲}
2. ...

【Open Questions】（あれば）
1. {未解決の判断}

次のアクション：
- 「レビューして」 → @eijient-architect-review --gate design-doc.md で設計レビュー
- 「実装して」 → @eijient design-doc.md を実装して
- 修正の指示 → 設計書を更新します
```

## 設計レンズ（必ず適用する）

1. **既存資産レンズ** — 何が既にこの問題を部分的に解いているか。新規コードより再利用。
   言語/フレームワークの標準機能・実績あるライブラリの探索を自作より先に行う
2. **境界と所有権レンズ** — コンポーネント境界は **Worker のファイル所有権に分解できる形**で切る。
   2人が同じファイルを触る設計は並列実装できない（設計時点で分担可能な境界にする）
3. **影経路レンズ** — データフローは happy / nil / empty / error の**4経路**を ASCII 図で描く。
   影経路の挙動が未定義の設計は未完成
4. **エラー命名レンズ** — 「エラーを処理する」で終わらせない。エラー名・トリガー・捕捉箇所・
   ユーザーに見える挙動を Error Map 表にする。沈黙の失敗ゼロ。catch-all は常に smell
5. **退屈技術レンズ** — イノベーション予算は約3枚。それ以外は実績ある退屈な技術をデフォルトにする
6. **可逆性レンズ** — feature flag・ロールバック手順・strangler fig を検討する。
   戻せない変更（DB migration・公開API変更）は設計書に明示する
7. **実装時判断の先取りレンズ** — 実装中に発生する判断を列挙し、いま解決する。
   解決できないものは Open Questions として顕在化させる（実装者に暗黙に委ねない）
8. **運用レンズ** — observability もスコープ（ログ・アラート・runbook は成果物）。
   疲れた人間が3amに運用できる設計か
9. **複雑度レンズ** — 新規8ファイル超・新サービス2つ超は smell。
   本質的複雑さと偶発的複雑さを区別し、後者は設計で消す

## design-doc と eijient ループの対応

design-doc.md は @eijient への入力インターフェース。各セクションはループでそのまま使われる：

| design-doc のセクション                  | ループでの使われ方                     |
| ---------------------------------------- | -------------------------------------- |
| Component Design（ファイル所有権マップ） | Planner の Worker 分担・担当ファイル指定 |
| 実装フェーズ                             | Planner の実装計画                     |
| Verification Requirements                | verifier のチェックリスト              |
| Error Map / Data Flow                    | tester のテストケース設計              |

---

## 引数

| 引数       | 短縮 | デフォルト      | 説明                                                     |
| ---------- | ---- | --------------- | -------------------------------------------------------- |
| `--output` | `-o` | `design-doc.md` | 設計書の出力ファイル名                                   |
| `--scope`  | `-s` | 全体            | 探索・設計対象を特定ディレクトリ/モジュールに限定        |
| `--review` |      | false           | 出力後そのまま @eijient-architect-review のレビューに続く |

## 使用例

```bash
# 基本：探索 → 設計案比較 → design-doc.md 出力 → 停止
@eijient-architect 通知システムを設計して

# 範囲を限定して設計
@eijient-architect --scope src/api 決済APIの設計をして

# 出力後そのままレビューまで
@eijient-architect --review リアルタイム同期機能を設計して

# 承認後の実装（@eijient に引き渡す）
@eijient design-doc.md を実装して
```

---

## 注意事項

- **実装しない。** `design-doc.md` の Write 以外の書き込みを禁止する
- Step 3 の質問に回答を得ずに設計を確定しない。回答が得られない前提で進める場合は、
  置いた仮定を design-doc の Open Questions に明示する
- 公開API・DB schema・認証・課金・外部連携に影響する設計判断は、必ず明確化質問に回す
- 探索は Explore サブエージェントに委譲し、メインコンテキストは設計判断に使う
