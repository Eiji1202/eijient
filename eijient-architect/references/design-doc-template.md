# design-doc.md テンプレート

このテンプレートに沿って、ルート直下の `design-doc.md` を生成する。
`{...}` は設計結果で埋める。各セクションは必須（該当なしの場合は「なし」と明記し、省略しない）。

人間がそのまま保存し、実装チームに次の形で渡せる内容にする：

> 「@eijient design-doc.md を実装して」

---

```markdown
# Design Doc: {機能名}

対象リポジトリ: {パス}
作成日: {日付} / 探索手法: {並列探索の分割（例: 既存資産 / パターン / 統合ポイント）+ スポット検証の有無}

## Objective

{この設計が可能にすることを1〜3行で。「誰のため・何を可能にするか」を含める。}

## Requirements & Constraints

- 機能要件: {...}
- 非機能要件: {パフォーマンス・セキュリティ・可用性など、明示されたもののみ}
- 制約: {既存システム・技術スタック・互換性の制約}

## 決定記録

検討した案と決定。3ヶ月後に「なぜこの設計か」が分かるように書く。

| 論点 | 採用 | 却下した案 | 理由 |
| ---- | ---- | ---------- | ---- |
| {設計判断} | {採用案} | {却下案} | {トレードオフと決め手} |

## Architecture

{コンポーネント間の関係を ASCII 図で。依存の向きを矢印で明示する。}

## Component Design（ファイル所有権マップ）

各コンポーネントは Worker 1人に割り当て可能な単位で切る。**担当ファイルが重複してはならない。**
共有ファイル（型定義など）は「Planner が先に作成」と明記する。

| コンポーネント | 担当ファイル | 責務 | 依存（先に必要なもの） |
| -------------- | ------------ | ---- | ---------------------- |
| {名前} | {glob（例: src/api/notifications/**）} | {...} | {...} |

## Data Flow

主要なデータフローを4経路すべて描く：

\```
[INPUT] → [VALIDATION] → [TRANSFORM] → [PERSIST] → [OUTPUT]
happy: {正常時の流れ}
nil:   {入力が欠けているとき何が起きるか}
empty: {入力が空のとき何が起きるか}
error: {上流が失敗したとき何が起きるか}
\```

## Error Map

「エラーを処理する」で終わらせない。すべてのエラーに名前をつける。

| エラー | トリガー | 捕捉箇所 | ユーザーに見えるもの | テストで検証するか |
| ------ | -------- | -------- | -------------------- | ------------------ |
| {エラー名/例外クラス} | {何が起こすか} | {どこで rescue/catch} | {表示・ログ・アラート} | {✅/⚠️} |

## Test Strategy

- ユニット: {対象と観点}
- 統合/E2E: {対象と観点。E2E が不要ならその理由}
- Error Map・影経路のうちテストで固定するもの: {...}

## 実装フェーズ

依存順に小さく切る。各フェーズは「明確な成果物を持つ単位」にする。

1. {フェーズ1（例: 共有型定義の作成 — Planner が先行）} — 検証: {方法}
2. {フェーズ2} — 検証: {方法}
3. ...

## Verification Requirements

実装完了の判定に使う、**実行可能なコマンド**。verifier がこのまま実行する。

| コマンド | 期待結果 |
| -------- | -------- |
| {例: npm test} | {例: 全件 PASS} |
| {例: npm run typecheck} | {例: エラー 0} |

## Open Questions

実装前に解決すべき未解決の判断。なければ「なし」。

1. {質問 — なぜ重要か — 回答によって設計のどこが変わるか}

## Out-of-scope

この設計に含めないもの（将来の拡張・意図的に外したもの）と理由。

- {項目} — {理由}
