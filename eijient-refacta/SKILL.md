# eijient-refacta Skill

## 概要

このSkillは「分析・計画」と「実行」を **明確に分離** したリファクタリングSkillです。

```
Phase 1: 分析  →  refactor-instructions.md をルートに出力  →  停止して承認を求める
                                                                      ↓ ユーザーが承認
Phase 2: 実行  ←  指示書のフェーズに沿って小さく安全にリファクタリング
```

**最重要ルール：**

1. **Phase 1 では絶対に実装・編集しない。** 読み取りと分析だけを行う。
2. 分析結果は必ず **ルートの `refactor-instructions.md`** に書き出す。
3. 書き出したら **必ず停止し、ユーザーに実行可否を確認する。**
4. **ユーザーが明示的に承認するまで Phase 2（実行）には進まない。**
5. 証拠（実際のファイル・コード）にもとづいて判断する。推測だけで負債を断定しない。

---

## ワークフロー

## Phase 1: 分析（実装しない）

### Step 1: 現在状態の確認

```bash
git status
```

未コミットの変更がある場合は、その内容を記録する。Phase 2 で自分の変更と混ぜないために重要。

### Step 2: プロジェクトの読み込み

存在するものを **すべて読む**。推測で飛ばさない。少なくとも以下を確認する：

- `AGENTS.md` / `CLAUDE.md` / `README` / プロジェクト固有の指示ファイル
- package / 依存関係の設定（`package.json`, `pyproject.toml`, `go.mod`, `Cargo.toml` など）
- build / lint / test / typecheck / CI 設定（`*.config.*`, `.github/workflows/`, `Makefile` など）
- docs / spec / 設計メモ
- エントリーポイント（`main`, `index`, `app`, サーバ起動箇所など）
- 主要モジュール
- テスト
- DB schema / migration
- 認証 / 課金 / 通知 / 外部API / ジョブ / キュー / ストレージ関連
- 既存スクリプト・運用手順

> 大規模な探索は、メインコンテキストを汚さないようサブエージェント（Explore / general-purpose）に任せてよい。
> ただし「最終判断」は証拠を自分で確認してから行う。

### Step 3: プロジェクト理解の整理

次を証拠とともに整理する：

- このプロジェクトは何をするものか
- 主要なユーザー体験・ワークフロー
- 主要なエントリーポイント
- 主要モジュールと責務
- データの流れ
- 外部依存
- 現在の検証コマンド（build / lint / test / typecheck）
- **絶対に壊してはいけない既存挙動**

### Step 4: 技術的負債の洗い出し

以下の観点でコードを精査する：

重複 / 死コード / 責務の混在 / 所有範囲の曖昧さ / 過剰または不足した抽象化 /
依存方向の乱れ / 型・schema・契約の曖昧さ / テスト不足・壊れやすいテスト /
エラーハンドリング・ログの不統一 / 設定・環境差分の複雑さ /
非同期・並行処理の危険箇所 / パフォーマンスのリスク /
セキュリティ上慎重に扱うべき境界 / 命名・ファイル配置の分かりにくさ

**各負債について、必ず以下を書く：**

| 項目                     | 内容                                       |
| ------------------------ | ------------------------------------------ |
| 根拠                     | ファイルパス・行・該当コード               |
| なぜ負債か               | 具体的な理由                               |
| 影響範囲                 | 影響するモジュール・機能                   |
| 変更リスク               | 触ったときに壊れうるもの                   |
| 改善案                   | 具体的な手段                               |
| 検証方法                 | 改善後に正しさを確認する方法               |
| 今実装してよいか / 提案のみ | 安全に実行可 か、承認が必要な提案に留めるか |

### Step 5: 「実装前に確認すべき質問」の抽出

コードから正しさを判断できないものは、**勝手に決めない。** 質問にする。

特に以下は **必ず質問** にする：

- 正しい仕様がコードから判断できない
- テストと実装が矛盾している
- 削除候補のコードが本当に不要かわからない
- 公開API・DB schema・保存済みデータに影響しうる
- 認証・課金・通知・外部連携に影響しうる
- 互換性を壊しうる
- 複数の設計案があり、プロダクト判断が必要

コードから明確に判断できるものは質問にしない（質問の乱発を避ける）。

### Step 6: refactor-instructions.md をルートに出力

`references/instruction-template.md` のテンプレートに沿って、ルート直下に `refactor-instructions.md` を **Write** する。

含めるセクション（すべて必須）：

```
Objective / Project Understanding / Behaviors To Preserve / Non-Negotiables /
Stop And Ask Conditions / Baseline Commands / Debt Map / Implementation Phases /
Verification Requirements / Reporting Format / Out-of-scope Items
```

**Implementation Phases は小さく安全な順番にする：**

1. 現在状態と検証コマンドを確認する
2. 重要挙動にテスト・再現手順がなければ先に安全網を作る
3. 明らかに安全な整理から始める
4. 小さな責務分離を行う
5. 境界・インターフェースを明確にする
6. テストしやすい構造にする
7. 大きな設計変更は、承認なしに実装せず提案に留める

### Step 7: 停止して承認を求める

`refactor-instructions.md` を出力したら、**実装に入らず停止** して次を報告する：

```
📋 リファクタ指示書を作成しました: refactor-instructions.md

【プロジェクト理解】
{1〜2行サマリー}

【検出した主な負債】（{N}件）
1. {負債} — {影響} — {今実装可 / 提案のみ}
2. ...

【実装前に確認すべき質問】（あれば）
1. {質問}
2. ...

【提案する実装フェーズ】
Phase 1: {...}
Phase 2: {...}
...

このまま Phase 2（実行）に進んでよいですか？
- 「実行して」 → 指示書に沿ってリファクタリングを開始します
- 質問への回答 → 指示書を更新します
- 「分析のみでいい」 → ここで終了します
```

**ユーザーが明示的に承認するまで、絶対に Phase 2 に進まない。**
「実装前に確認すべき質問」がある場合は、回答を得て指示書を更新してから承認を求める。

---

## Phase 2: 実行（承認後のみ）

ユーザーが承認したら、`refactor-instructions.md` の内容に従って実行する。

### 実行時の制約（必ず守る）

- 最初に `git status` を確認する
- 既存の未コミット変更と自分の変更を混ぜない
- 編集前に baseline の検証結果（build / lint / test）を記録する
- 変更は **小さく戻しやすい単位** にする（フェーズごとにコミット）
- 無関係な整形・ついでのリファクタリングをしない
- 既存挙動を勝手に変えない
- 正しさが不明になったら **実装を止めて質問する**
- 各フェーズごとに検証する（指示書の Verification Requirements に従う）
- 大きな設計変更（Out-of-scope / 提案のみ のもの）は実装しない
- 最後に、実行したコマンドと結果を報告する

### 実行方法の選択

- **規模が大きい / 並列で進められる** 場合は `@eijient` にスケール指定で委譲してよい
  （例: `@eijient --plan-approval refactor-instructions.md の Phase 1〜3 を実行して`）
- **小規模 / 逐次的** な場合はこのSkill内で直接実行する

どちらの場合も「フェーズごとに検証 → コミット → 次のフェーズ」を厳守する。

### 完了報告

```
✅ リファクタリング完了（または途中報告）

【実行したフェーズ】
- Phase 1: {内容} — {検証結果}
- Phase 2: {内容} — {検証結果}

【実行したコマンドと結果】
- {コマンド}: {結果}

【未実施・提案に留めたもの】
- {Out-of-scope の項目とその理由}

【次にやるべきこと】（あれば）
- {残課題}
```

## 引数

| 引数             | 短縮 | デフォルト | 説明                                                       |
| ---------------- | ---- | ---------- | ---------------------------------------------------------- |
| `--analyze-only` | `-a` | false      | Phase 1 のみ実行（指示書を出力して必ず終了。承認確認なし） |
| `--execute`      | `-e` | false      | 既存の `refactor-instructions.md` から Phase 2 を実行する  |
| `--output`       | `-o` | `refactor-instructions.md` | 指示書の出力ファイル名                        |
| `--scope`        | `-s` | 全体       | 分析・リファクタ対象を特定ディレクトリ/モジュールに限定    |

---

## 使用例

```bash
# 基本：分析 → 指示書出力 → 承認待ち
@eijient-refacta このプロジェクトのリファクタ計画を立てて

# 範囲を限定して分析
@eijient-refacta --scope src/api API層の技術的負債を洗い出して

# 分析だけして終わる（実行しない）
@eijient-refacta --analyze-only リファクタ指示書だけ作って

# 既存の指示書からリファクタを実行する
@eijient-refacta --execute
```

## ベストプラクティス（必ず守ること）

### 分析と実装を混ぜない

- Phase 1 で一切ファイルを編集しない。`refactor-instructions.md` の Write だけが唯一の書き込み。
- 「ついでに直す」をしない。発見した負債はすべて指示書に記録し、実装は承認後に行う。

### 証拠主義

- 「たぶん使われていない」で死コードを断定しない。参照箇所を grep で確認する。
- 古いコード = 悪 と決めつけない。動いている既存挙動は守るべき仕様として扱う。

### 安全な順番を守る

- テストや再現手順がない重要挙動は、リファクタ前に **先に安全網（テスト）** を作る。
- 明らかに安全な整理 → 小さな責務分離 → 境界の明確化 の順に進める。
- 大きな設計変更・全面書き換えは、承認なしに実装しない（指示書では「提案のみ」に分類する）。

### スコープを膨らませない

- 「全部リファクタして」のような曖昧な計画にしない。
- 見た目の綺麗さを目的にしない。目的は「既存仕様を壊さず、負債を減らし、変更しやすくする」こと。

---

## 注意事項

- **Phase 1 では絶対に実装しない。** 指示書の Write 以外の編集を禁止する。
- 指示書を出力したら必ず停止し、ユーザーの承認を得てから Phase 2 に進む。
- `--analyze-only` 指定時は承認確認もせず、指示書出力後に終了する。
- 削除候補・公開API・DB schema・認証/課金/外部連携に関わるものは、必ず「実装前に確認すべき質問」に回す。
- 実行フェーズでは各フェーズごとに検証し、小さく戻しやすい単位でコミットする。
