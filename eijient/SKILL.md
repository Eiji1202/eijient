# eijient — Loop-First Agent Team Skill

## 意図

開発タスクを Agent Team の自律ループで完遂する。ユーザーが見ていない間（夜間運用）でも
「計画 → 実行 → 検証 → 記録」を回し、**検証済みの成果だけ**を PR として朝のレビューに残す。

達成すべき成果:

- GitHub Issue（完了条件つき）→ 作業ブランチ → 実装 → 検証 → PR まで無人で到達する
- 進捗・完了の報告はすべて証拠つき（このセッションのツール結果を指せるものだけ）
- 得られた教訓を対象プロジェクトのメモリに記録し、次の周回に引き継ぐ

## ループ構造

1. **計画** — Planner がコードベースを調査し、設計・タスク分解・完了条件を決めて Worker に共有する
2. **実行** — 各 Worker が担当領域を実装する
3. **検証** — verifier（実装に関与していないまっさらなコンテキスト）が動作を証拠つきで確認する
4. **記録** — 教訓をメモリに記録する。未完なら修正して次の周回へ、完了なら PR を作成して停止

## チーム構成

規模は自動判定する（迷ったら medium）。構成の詳細は `references/templates.md`、
spawn プロンプトは `references/prompts.md` を使う。

| スケール   | 判定基準                                 | 構成                                  |
| ---------- | ---------------------------------------- | ------------------------------------- |
| **small**  | 単一機能・単一ファイル変更・バグ修正     | Planner + Worker 1（検証は単発サブエージェント） |
| **medium** | 複数ファイル・複数レイヤー・新機能実装   | Planner + Worker 2〜3 + verifier      |
| **large**  | 複数機能・フルスタック・大規模リファクタ | Planner + Worker 3〜5 + verifier      |

## Spawn プロンプトに必ず埋め込むもの

詳細な文言は `references/prompts.md` の共通ブロックを使う。

- **意図の受け渡し** — 依頼文だけを渡さない。「私は[大きなタスク]を[誰のため]にやっている。
  彼らには[アウトプットが可能にすること]が必要。それを踏まえて: [依頼]」の形で背景ごと渡す
- **証拠ベース報告** — 各主張をこのセッションのツール結果と照合し、証拠を指せる作業だけ報告する。
  未検証のものは「未検証」と明示する
- **anti-early-stop** — ユーザーはリアルタイムで見ていない。可逆的なアクションは聞かずに進める。
  ターンを終える前に最後の段落を確認し、それが計画・質問・約束なら、いま実行する。
  残りトークン数・コンテキスト残量には言及させない（早期切り上げを誘発する）
- **過去の教訓** — 対象プロジェクトのメモリからタスクに関連する教訓を選んで含める。
  メモリを自動で読み込むのはメインセッションだけで、spawn されたエージェントには届かない

## 境界（違反してはならないこと）

- **マージは絶対に行わない。** PR 作成と報告で停止する
- **ファイル所有権を守る。** 2体の Worker が同じファイルを編集すると Git コンフリクトではなく
  **無言の上書き**になる。Worker ごとに担当ファイルを明確に分け、共有ファイル（型定義など）は
  Planner が先に作成してから Worker に渡す
- **Planner は実装しない。** 設計・調整・統合に専念させる
- **完了報告を鵜呑みにしない。** `git diff --name-only HEAD` で実変更を確認してから次へ進む
- **GitHub 操作（Issue / PR）は Team Lead のみが行う**
- Issue には完了条件（Acceptance Criteria）を書き、PR 本文には `Closes #{Issue番号}` を書く。
  ブランチ名は `feat/issue-{番号}-{短い説明}`。コミットメッセージ・ブランチ運用は
  `~/.claude/rules/git-workflow.md` に従う

## 自律リカバリ（無人運用のための挙動）

Worker の停止・暴走にユーザーの介入を求めない:

1. 長時間報告のない Worker には進捗を要求し、応答がなければ `git diff` で実作業の証拠を確認する
2. 証拠がなければ Worker を再起動するか、タスクを他の Worker または Team Lead 自身に再割当する。
   **同一タスクの再起動・再割当は合計2回まで。** 超えたら未完了として記録し、次のタスクへ進む
3. 進捗の証拠が増えないまま周回が2回続いたら実行全体を打ち切り、完了分と未完了分を明示して報告する
4. インシデントと対処はメモリに記録し、最終報告で必ず開示する
5. `git push` が許可されない環境では、commit と差分の提示までで停止し、push・PR 作成は
   ユーザーの判断に委ねる（権限設定は変更しない）

## メモリ（教訓の記録）

対象プロジェクトの標準メモリディレクトリ（Claude Code がセッション開始時に読み込む
`~/.claude/projects/<プロジェクト>/memory/`）に記録する。tasks/ ディレクトリは作らない:

- 1ファイル1教訓。冒頭に1行サマリを書き、MEMORY.md の索引に1行追加する
- 同じ教訓は新規ファイルを作らず既存ファイルを更新する
- 間違いと判明した教訓は削除する
- 2周回以上で効いた教訓は、対象プロジェクトの CLAUDE.md や該当 Skill 本体に昇格させる
  （メモリを「記録されたが二度と浮上しない墓場」にしない）

## 完了時の報告

PR 作成後、証拠つきで報告して停止する:
Issue 番号 / PR 番号 / ブランチ / 変更ファイル一覧 / verifier の検証結果と差し戻し回数 /
発生したインシデントと対処。
差し戻し0回で全通過した場合はその旨を明記する（検証ゲートが機能していないサインの可能性を
ユーザーが監視できるようにする）。

## 引数

| 引数              | 短縮 | デフォルト | 説明                                          |
| ----------------- | ---- | ---------- | --------------------------------------------- |
| `--scale`         | `-s` | auto       | チームスケール: small / medium / large / auto |
| `--model`         | `-m` | adaptive   | モデル戦略: deep / adaptive / fast / budget   |
| `--plan-approval` | `-p` | false      | Workerの実装前にPlannerの設計承認を要求       |
| `--dry-run`       | `-d` | false      | チーム構成の確認のみ（spawnしない）           |
| `--no-issue`      |      | false      | GitHub Issue作成をスキップ                    |

### モデル戦略

| 戦略       | Planner | Worker | 用途                           |
| ---------- | ------- | ------ | ------------------------------ |
| `deep`     | fable   | sonnet | 最高品質が必要な複雑タスク     |
| `adaptive` | opus    | sonnet | 品質とコストのバランス（推奨） |
| `fast`     | sonnet  | sonnet | 速度重視・明確なタスク         |
| `budget`   | sonnet  | haiku  | シンプルなタスク・コスト重視   |

teammate のモデルに fable を指定できない環境では、`deep` の Planner を opus に落として続行する。

## 使用例

```
# 基本的な使い方（スケール自動判定）
@eijient ログイン機能を実装してください

# スケール指定
@eijient --scale large 認証システム全体をリファクタリングしてください

# 設計承認あり・dry-run確認
@eijient --plan-approval --dry-run Stripe決済を実装してください

# コスト重視
@eijient --model budget READMEを更新してください
```

## 注意事項

- Agent Teams は実験的機能。`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` が必要
- チームは 3〜5 体がベスト。それ以上はコスト増に見合わない。Worker 1体あたり最大5〜6タスク
- タスクの粒度は「関数・コンポーネント・テストファイルなど明確な成果物を持つ単位」に切る
- トークンコストは Worker 数 × セッション分かかる
- `gh` CLI がインストール・認証済みであること
