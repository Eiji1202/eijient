# Lessons

## 2026-07-03: Skillプロンプトには「手順」だけでなく「視点（レンズ）」を書く

- **背景**: eijient-refacta を Fable と Opus で同一リポジトリに実行して比較した。プロセス指示（読む→整理→出力）は同じでも、成果物の質は「どんな視点で見るか」で大きく差が出た。
- **教訓**: Skillプロンプトの品質は、ワークフローの正確さより「分析レンズの明文化」で決まる。優れた出力から差分となった視点を抽出し、プロンプトに還元するループを回す。
- **抽出したレンズ**（eijient-refacta に反映済み）:
  1. 時間軸 — git log / 移行計画docから「進行中プロジェクトの残骸」として負債を分類。計画doc自体も監査する
  2. 横展開漏れ — 直近コミットの強化パターンの適用漏れを探す
  3. 対照 — 同種の境界を比較し、片方にだけある防御を疑う
  4. 全数調査 — 「N箇所中M件（全数調査済み）」の形で報告。経路は全列挙
  5. リポジトリ外 — 兄弟リポジトリ・外部設定まで呼び出し元を探し、「確認できない事実」を質問化
  6. 運用衛生 — CVE番号・PIIハードコード・ローカルとCIのツール差分・本番到達可能なサンプルルート
  7. baseline実測 — 検証コマンドは分析時に実行し実測値を記録。REDもbaselineとして扱う
  8. red→green安全網 — 修正前に赤になるテストを書き、挙動変化を証拠化
  9. 質問は少なく・重く・優先度順 — 実装判断で足りるものは質問にしない
  10. 成果のゲート化 — クリーン化した検査はCIゲートに変換して恒久化
- **Opus由来でも採用した強み**: Behaviors To Preserve は挙動名でなく「何がそれを守っているか」（機構）を行番号つきで書く。近接する類似実装間の意味のある乖離（片方だけガードがない）を検出する。

## 2026-07-04: 記憶から復元した「検討中」の決定を確定事項として扱わない

- **背景**: 前セッションの実装方針を claude-mem から復元した際、記録に「リポジトリを eijient-loop にリネーム予定」とあったためプランに含めて実行しようとしたが、ユーザーの実際の意向は「リポジトリ名は eijient のまま」だった。セッションタイトルは「whether to rename（リネームするかどうか）」で、まだ検討段階だったことを示していた。
- **教訓**: 復元した記憶の中の決定事項は、記録の強い表現（"planned" 等）より弱いシグナル（"whether"・"considering"）を優先して解釈する。特に対外的・名前系の決定（リポジトリ名・公開URL・スキル名）は、計画に含める場合でも独立した確認項目として明示し、他の確定事項と一括承認にしない。
- **ルール**: 記憶復元ベースで計画を作るときは、各項目に「確定 / 検討中」のラベルを付け、検討中の項目は AskUserQuestion で個別確認してからプランに昇格させる。
